core principles

15 synthesized from 34 talks — the consensus that emerged when you strip away the marketing
Start with the simplest possible agent architecture
Begin with an augmented LLM (model + tools + retrieval) and only add agentic loops, multi-agent coordination, or framework abstractions when you hit a concrete bottleneck. Most production agents are just a prompt with an LLM call in a while loop — not framework-heavy architectures. If you find yourself specifying exact steps in the prompt, you should have written a deterministic workflow instead.
✗ avoid: Reaching for multi-agent frameworks, complex orchestration, or autonomous agents before proving a single well-prompted agent with tools can't solve the problem.
— Barry Zhang, Anthropic, Dex Horthy, HumanLayer, Luke Alvoeiro, Factory, Anju Kambadur, Bloomberg, Sayash Kapoor
Own your prompt, context window, and control flow
Never let a framework construct your prompt, decide what enters your context window, or control when the model gets called. You need to see and control every token going to the model, compact/summarize aggressively to prevent context rot, and keep control flow in simple while loops and if statements you can debug.
✗ avoid: Using framework abstractions that hide prompt construction, auto-manage context, or control the agent loop — then spending hours debugging 7 layers deep in a call stack.
— Dex Horthy, HumanLayer, Ryan Lopopolo, OpenAI, Patrick Debois, Tessl
Think like your agent — empathy-driven debugging
Trace through exactly what the agent sees at each decision point. If you couldn't solve the task given the same context, fix the context before blaming the model. Read agent traces line-by-line — there is no substitute. Close your eyes and try to navigate a webpage opening them every 10 seconds; that's what the model experiences.
✗ avoid: Debugging agents by looking at final outputs rather than tracing the full context the model received at each step.
— Barry Zhang, Anthropic, Ash & Andrew, Anthropic Applied AI, Dex Horthy, HumanLayer
Build evals as unit tests from day one
Evals require exactly three ingredients: a task function, a dataset of input/expected-output pairs, and at least one automated score. Use LLM-as-judge for fuzzy assertions, but validate the judge itself against human judgment. Score individual pipeline steps, not just end-to-end output. Run evals in CI/CD with threshold-based gates.
✗ avoid: Vibes-based evaluation ('this output looks better'), testing only final outputs, or skipping evals until production problems emerge.
— Dex Horthy, HumanLayer, Doug Guthrie, Braintrust, Anju Kambadur, Bloomberg, Sayash Kapoor
Context is the new code
As code generation becomes automated, what you author is context: specifications, skills, architectural decisions, conventions, and constraints. Code is a disposable build artifact compiled by the LLM from context. Invest in context management (versioning, testing, deploying) with the same rigor you apply to code.
✗ avoid: Treating prompts and context files as informal artifacts while applying rigorous engineering only to the generated code.
— Patrick Debois, Tessl, Ryan Lopopolo, OpenAI, Maggie Appleton, GitHub, Dex Horthy, HumanLayer
Treat multi-agent systems as distributed systems
Going from 1 agent to 5 agents is not adding features — it's building a distributed system. You inherit all distributed systems problems: race conditions, coordination overhead, state management, and cascading failures. Apply established patterns: event sourcing, circuit breakers, idempotent operations, and dead letter queues.
✗ avoid: Adding more agents to a system as if they're just additional function calls, without designing for coordination, failure recovery, and state consistency.
— Sandipan Bhaumik, Luke Alvoeiro, Factory, Anju Kambadur, Bloomberg
Reliability degrades multiplicatively — design for compound failure
If each step in a multi-step agent is 95% reliable, a 10-step agent is only ~60% reliable. Every additional step, tool call, or agent handoff multiplies your failure rate. Design for graceful degradation, implement fallback paths, and add human-in-the-loop checkpoints at high-risk decision points.
✗ avoid: Evaluating agent reliability based on individual step success rates without modeling the compound probability across the full trajectory.
— Anju Kambadur, Bloomberg, Douwe Kiela, Contextual AI, Randall Hunt, Caylent
Build skills, not agents
Instead of building separate agents per domain, build one universal code-executing agent and equip it with modular, reusable skill packages. Skills encode domain expertise (task instructions, tool configurations, domain patterns) and use progressive disclosure — load only metadata initially, expand full instructions when activated. Customization happens at the skill layer, not the agent architecture layer.
✗ avoid: Building N separate agents with different architectures for N domains instead of one capable agent with N skill modules.
— Barry Zhang & Mahesh Murag, Anthropic, Thariq Shihipar, Anthropic, Patrick Debois, Tessl
Spec-driven development is the new engineering work
The ratio of planning to implementation is inverting: expect 80% planning and specification, 20% coding. Agents need unambiguous specifications. This forces teams to resolve disagreements upfront rather than during implementation. Writing specs IS the engineering work — and if your agent can't figure it out from your specs, neither can a new engineer.
✗ avoid: Skipping specification to 'just let the agent code it' — producing inconsistent, conflicting output that requires more rework than writing specs would have taken.
— Maggie Appleton, GitHub, Ryan Lopopolo, OpenAI, Martin & Natasha, McKinsey Software X, Mike Spitz, PFF
Use the file system as your context engineering surface
The file system is the most powerful context management primitive. Save tool results to files, create memory folders, use JSON for persistent state (models overwrite markdown but respect JSON structure), and design CLIs with --help for progressive agent discovery. Your agent file should be ~50 lines, but the file system context around it is where the real work lives.
✗ avoid: Keeping everything in the conversation context instead of externalizing state, results, and working memory to the file system.
— Thariq Shihipar, Anthropic, Ash & Andrew, Anthropic Applied AI, Luke Alvoeiro, Factory, Ryan Lopopolo, OpenAI
Separate generator and evaluator with independent contexts
Inspired by GANs: split builder and critic into different context windows with different system prompts. The evaluator should test actual output (run the app with Playwright, execute the code) not just read diffs. Never share context between generator and evaluator — the model 'kids itself' when evaluating its own work in the same context.
✗ avoid: Having the same agent in the same context both generate and evaluate its own output — producing sycophantic self-assessment.
— Ash & Andrew, Anthropic Applied AI, Thariq Shihipar, Anthropic, Ryan Lopopolo, OpenAI
Fail loudly and visibly
Surface errors visibly rather than silently retrying or swallowing exceptions. Every time you type 'continue' to an agent is a failure of the harness to provide enough context. Agents should fail predictably (deterministic failure modes you can debug) rather than succeed unpredictably (non-reproducible results you can't trust).
✗ avoid: Silently retrying failed operations, swallowing exceptions, or building agents that produce inconsistent results that seem correct but can't be reproduced.
— Dex Horthy, HumanLayer, Ryan Lopopolo, OpenAI, Ash & Andrew, Anthropic Applied AI
Optimize for agent productivity, not just human productivity
The bottleneck has shifted from human code output to agent throughput. Reframe the optimization target: structure repos for agent locality (isolated packages, one canonical way to do everything), invest in context engineering (CLAUDE.md, scoped tasks), and eliminate patterns that cause agent-generated slop through systematic 'Garbage Collection' sessions.
✗ avoid: Optimizing only human developer workflows while agents struggle with inconsistent repo structures, missing context, and ambiguous conventions.
— Mike Spitz, PFF, Ryan Lopopolo, OpenAI, Dex Horthy, HumanLayer, Dan Shipper, Every
MCP is the integration standard — build for it
Model Context Protocol solves the N×M integration problem: build your MCP server once and it works with every MCP client (Claude, ChatGPT, VS Code, Cursor). Three control planes: tools (model-controlled), resources (application-controlled with subscriptions), prompts (user-triggered). The protocol is evolving to include remote servers with OAuth, registries for dynamic tool discovery, and dual human+agent interfaces.
✗ avoid: Building custom, one-off tool integrations for each AI application instead of implementing the MCP standard once.
— Mahesh Murag, Anthropic, David Soria Parra, Anthropic, Barry Zhang & Mahesh Murag, Anthropic
RAG is not solved after the demo works
Basic top-K vector search is just the starting point. Production RAG requires hybrid retrieval (vector + BM25 + metadata filtering), reranking with cross-encoders, end-to-end pipeline optimization (not component-by-component), grounded generation with source attribution, and the ability to say 'I don't know.' Optimize as a system, not individual components.
✗ avoid: Treating RAG as solved after a basic vector-search demo works, then being surprised when it fails on production queries with ambiguity, multi-hop reasoning, or domain-specific terminology.
— Jerry Liu, LlamaIndex, Douwe Kiela, Contextual AI

situational playbook

when you face X, do Y, avoid Z — 30 plays organized by domain

[agent design]

When you're deciding whether to build an agent or a deterministic workflow...
→ do: Ask: does this task require dynamic decision-making based on environment feedback? If you can specify exact steps, write a deterministic workflow (bash script, DAG). Only use an agent when the path depends on intermediate results.
Agents add latency, cost, and unpredictability — they're only worth it when the task genuinely requires adaptive behavior.
✗ avoid: Building agents for tasks where you end up specifying every step in the prompt anyway — you could have written the script in 90 seconds.
— Barry Zhang, Anthropic, Dex Horthy, HumanLayer
When you're architecting a new agent system...
→ do: Start with a single well-prompted agent with tools. Measure where it fails. Decompose into multi-agent only when you hit clear bottlenecks in context window size, capability specialization, or parallelism needs.
Every agent boundary adds coordination overhead and failure modes — add complexity only when single-agent limitations are proven.
✗ avoid: Designing a multi-agent system from the start because it seems more 'sophisticated' or because a framework makes it easy to add agents.
— Luke Alvoeiro, Factory, Barry Zhang, Anthropic, Sandipan Bhaumik
When your agent needs to run for hours on complex tasks...
→ do: Break the task into a feature list. Execute each in a fresh context window. Use file-system state (JSON progress files) for continuity. Implement Generator-Evaluator: separate builder and critic contexts, with the evaluator testing actual output via Playwright/puppeteer. Git commit after each verified feature.
Context rot degrades coherence in long sessions, and models become sycophantic about their own output when evaluating in-context.
✗ avoid: Running everything in one growing session, sharing context between generator and evaluator, or using markdown for progress state (models overwrite it).
— Ash & Andrew, Anthropic Applied AI
When you need agents to handle real customer interactions...
→ do: Treat agent development as experience design, not engineering. Design for the full complexity: edge cases, emotional users, multi-step processes, and graceful human hand-offs. Staff the 'AI Architect' role — someone combining product thinking, conversation design, domain knowledge, and technical skills.
Demo-quality is not production-quality. The 80% long tail of edge cases is where trust is built or lost.
✗ avoid: Treating customer-facing agents as a pure engineering problem, or launching based on demo performance without testing the full distribution of real interactions.
— Clay Bavor, Sierra
When you're choosing between open and closed models for agents...
→ do: Consider open models when privacy, predictability, and cost control matter. Open weights enable quantization, fine-tuning, edge deployment, and guarantee no silent performance degradation. Use closed frontier models when you need maximum capability and can tolerate API dependency.
Open models have reached parity on many benchmarks, and with closed models, performance can degrade without notification.
✗ avoid: Assuming closed models are always better, or assuming open models can't handle production agent workloads.
— Merve Noyan, HuggingFace, Anju Kambadur, Bloomberg
When your agent needs to handle plans that evolve...
→ do: Implement plans as living documents (mutable artifacts) that agents revise as they discover new information. Break into features, execute each in a fresh context, loop until all pass tests. Use Opus-class models for planning and Sonnet-class for execution.
Static plans fail when agents discover the problem is different than assumed. Fresh contexts per sub-task prevent context degradation over long runs.
✗ avoid: Treating plans as immutable specifications, or running all planning and execution in a single growing context.
— Luke Alvoeiro, Factory, Ash & Andrew, Anthropic Applied AI
When you're deciding how to structure your repo for AI agents...
→ do: Structure like a 10,000-engineer org even if you're small: isolated packages by domain, one canonical way to do everything (one ORM, one async helper, one CI style). Create curated context files (CLAUDE.md, AGENTS.md). Design CLIs with --help for progressive agent discovery. Put scripts in the file system and reference them in system prompts.
Agent output becomes predictable when the repo has consistent patterns regardless of where the agent looks — inconsistency is the primary driver of agent slop.
✗ avoid: Having multiple ways to do the same thing in different parts of the codebase, or relying on tribal knowledge that isn't captured in files the agent can read.
— Ryan Lopopolo, OpenAI, Thariq Shihipar, Anthropic, Dex Horthy, HumanLayer

[multi-agent]

When you're choosing a multi-agent coordination pattern...
→ do: Match the pattern to the task: Delegation for simple subtasks (parent spawns child, waits). Handoff for sequential pipelines. Broadcast/fan-out for verification and consensus. Orchestration for complex multi-step projects. Choreography (event-driven, pub/sub) for loose coupling and independent scaling.
Each pattern has different tradeoffs for coupling, scalability, and failure handling — using the wrong one creates unnecessary complexity.
✗ avoid: Defaulting to full orchestration for every multi-agent scenario, or building peer-to-peer autonomous teams before mastering simpler patterns.
— Luke Alvoeiro, Factory, Sandipan Bhaumik

[memory & context]

When you're designing agent memory...
→ do: Design memory as a first-class architectural concern with at least three layers: working memory (current session), long-term memory (persistent cross-session storage via database), and procedural memory (learned patterns encoded in skills/prompts). Combine vector similarity search with time-stamped logs for richer recall.
Memory transforms transactional interactions into ongoing relationships and reduces prompt engineering burden by storing context dynamically instead of cramming it into system prompts.
✗ avoid: Bolting memory onto an agent as an afterthought, or using only conversation history as 'memory' without persistent, queryable storage.
— Richmond Alake, MongoDB
When you're managing context for a coding agent...
→ do: Compact and summarize aggressively. Use sub-agents to do work in isolated contexts, returning only results to the main agent. Save tool call results to files. Surface instructions to the model at the right time — defer detailed context to lint/test time so the agent prototypes first, then refines.
Context rot degrades coherence — every wasted token in the context window reduces the model's ability to reason about the actual task.
✗ avoid: Front-loading all instructions into the system prompt, keeping everything in conversation context, or letting the context window grow without compaction.
— Thariq Shihipar, Anthropic, Ryan Lopopolo, OpenAI, Dex Horthy, HumanLayer

[evals & testing]

When you're setting up evals for an AI feature...
→ do: Define the eval triad: task function, dataset of input/expected-output pairs, and at least one automated score. Score individual pipeline steps (not just final output). Build the flywheel: production logs → human review → add interesting cases to eval dataset → better prompts → production.
Span-level scoring catches root causes (e.g., a bad query rephrasing step tanks everything downstream), and the flywheel ensures evals improve over time.
✗ avoid: Evaluating only final outputs end-to-end, using LLM-as-judge scores as absolute (they're relativistic — for comparing iterations), or skipping validation of the judge itself.
— Doug Guthrie, Braintrust, Dex Horthy, HumanLayer
When you're building an agent's verification loop...
→ do: Add deterministic verification at every possible step: linting, compilation, null checks, file-read-before-write guards. After implementing a feature, have the agent actually run it, test it visually (Playwright), and verify before marking complete. Use sub-agents for adversarial checking.
Verification should happen everywhere, not just at the end — catching errors at intermediate steps prevents cascade failures and wasted context.
✗ avoid: Deferring all verification to a final check, or having the agent verify its own work in the same context where it generated it.
— Thariq Shihipar, Anthropic, Ash & Andrew, Anthropic Applied AI
When you need to debug why an agent is producing poor output...
→ do: Read the full trace line-by-line, focusing on what context the model received at each decision point. Check for context rot (is important info being pushed out?), wrong tool granularity (is the model getting too much or too little from tools?), and ambiguous instructions. Fix the context before tweaking the model.
Agent failures are almost always context failures. If a human couldn't solve the task with the same context, no model can either.
✗ avoid: Jumping to model swaps, temperature tuning, or prompt hacks before understanding what the model actually saw when it made bad decisions.
— Barry Zhang, Anthropic, Ash & Andrew, Anthropic Applied AI, Dex Horthy, HumanLayer
When you're measuring AI coding tool impact on your team...
→ do: Measure actual value (features shipped, bugs fixed, business impact) not output volume. Use composite metrics including code quality and review cycle time. Expect 20-30% gains on specific task types, not 10x across the board. Measure on private repos — public repo studies are fundamentally flawed.
High output with high rework is negative value. Lines of code is a terrible metric — you might be 'shipping a lot more but much of it is reworking the slop you shipped last week.'
✗ avoid: Using LoC or PR count as productivity metrics, believing 10x vendor claims, or judging AI tool effectiveness from public repo studies.
— Yegor Denisov-Blanch, Stanford, Dex Horthy, HumanLayer

[production]

When you're deploying an agent to production...
→ do: Start with deterministic workflows and selectively replace steps with LLM-driven components. Build explicit guardrails, fallback paths, and human-in-the-loop checkpoints. Measure per-step reliability and model compound failure rates. Set latency budgets early.
The POC-to-production gap is about reliability, error handling, and cost — not model capability. Users abandon slow systems regardless of quality.
✗ avoid: Big-bang agent deployment, assuming POC reliability will hold at scale, or ignoring latency until after quality is optimized.
— Anju Kambadur, Bloomberg, Randall Hunt, Caylent, Douwe Kiela, Contextual AI
When you're using AI coding agents on a real (brownfield) codebase...
→ do: Invest heavily in context engineering: curate CLAUDE.md / AGENTS.md files, scope tasks tightly to relevant code sections, use strategic file inclusion. Structure repos for agent locality — isolated packages by domain, one canonical way to do everything.
A team got 2-3x more productivity from Claude Code after cracking context engineering — the default experience dramatically underperforms on brownfield codebases.
✗ avoid: Exposing the full codebase to the agent without scoping, or judging AI coding tools based on the out-of-box experience without investing in context.
— Dex Horthy, HumanLayer, Ryan Lopopolo, OpenAI
When you're implementing agent security...
→ do: Apply Swiss cheese defense: model alignment + harness permissioning (AST parser on bash tool to block dangerous patterns) + sandboxing (network + filesystem isolation). Watch for the 'lethal trifecta': execute + filesystem access + exfiltration capability.
No single layer is sufficient — each layer has holes, but stacking them makes exploitation require bypassing all layers simultaneously.
✗ avoid: Relying solely on prompt-level safety instructions or solely on sandboxing without defense in depth.
— Thariq Shihipar, Anthropic
When you're deploying agents at enterprise scale...
→ do: Focus on vertical-specific agents with clear ROI metrics (customer support, HVAC optimization, financial analysis). Start with high-volume, low-complexity tasks to prove reliability. Build incrementally: copilot mode → semi-autonomous → autonomous. Invest in domain-specific evaluation with expert annotators.
Only 1 in 4 businesses get value from AI — the gap is context and process, not capability. Vertical agents with measurable ROI actually ship.
✗ avoid: Deploying general-purpose autonomous agents or chasing 'moonshot' use cases when proven ROI exists in support, knowledge management, and content generation.
— Joao Moura, CrewAI, Anju Kambadur, Bloomberg, Randall Hunt, Caylent, Douwe Kiela, Contextual AI, NLW
When you're allocating token budget across an agent pipeline...
→ do: Split roughly 1/3 planning+tickets, 1/3 implementation, 1/3 CI/verification. Spending tokens on CI is essential because getting code accepted is harder than writing it. Use skills that are narrow (5-10) and deep rather than wide.
Most teams under-invest in verification tokens — they generate code cheaply but waste human time on review when automated checks could have caught issues.
✗ avoid: Spending 90% of token budget on generation and 10% on verification — the ratio should be much more balanced.
— Ryan Lopopolo, OpenAI
When your agent calls tools that can fail...
→ do: Make every agent operation idempotent. Implement circuit breakers for repeated failures, dead letter queues for unprocessable messages, and graceful degradation when tools are unavailable. The agent should degrade to best-effort rather than crash or hallucinate.
Tool-use reliability is the biggest pain point in production agents — LLMs frequently hallucinate tool calls, use wrong parameters, or chain tools incorrectly.
✗ avoid: Assuming tools will always succeed, or having the agent hallucinate results when a tool call fails instead of explicitly handling the failure.
— Sandipan Bhaumik, Anju Kambadur, Bloomberg, Douwe Kiela, Contextual AI

[rag]

When you're building RAG for production...
→ do: Implement hybrid retrieval (vector + BM25 + metadata filtering), add a reranking step with cross-encoders, and build separate evaluation pipelines for retrieval quality (precision/recall of chunks) and generation quality (faithfulness/relevance). Tune chunk size and overlap as hyperparameters per use case.
Basic top-K vector search works for demos but fails on production queries with ambiguity, domain jargon, or scale.
✗ avoid: Stopping at basic vector search, treating chunk size as one-size-fits-all, or evaluating only the final answer without measuring retrieval quality independently.
— Jerry Liu, LlamaIndex, Douwe Kiela, Contextual AI
When enterprise users need to trust your RAG system...
→ do: Implement grounded generation with source attribution for every claim. Build the 'I don't know' capability — systems that confidently refuse when context is insufficient are more trusted. Combine fine-tuning (for domain style/format) with RAG (for factual grounding).
Hallucination tolerance in enterprise is near zero — a system that makes one confident but wrong claim loses trust permanently.
✗ avoid: Building systems that always produce an answer even when context is insufficient, or relying solely on RAG without domain fine-tuning for style and terminology.
— Douwe Kiela, Contextual AI, Jerry Liu, LlamaIndex
When you're automating knowledge work with agents...
→ do: Identify repetitive document-heavy workflows (report generation, data extraction, document comparison) as first targets. Use DAG-based workflow agents for structured processes. Design UX as task queues with review interfaces, not chat. Start with assistive agents producing drafts for human review.
90% of enterprise data is unstructured documents. Agent-as-chatbot UX doesn't fit knowledge work — agents are workers, not conversational partners.
✗ avoid: Building all knowledge work agents as chatbots, or deploying fully autonomous agents on high-stakes document workflows without human review gates.
— Jerry Liu, LlamaIndex

[team & org]

When you need your team's agents to produce consistent code...
→ do: Build a shared context layer: architectural decision records, conventions, constraints, and code style guides accessible to ALL agents on the team. Institute Garbage Collection sessions to categorically eliminate recurring agent slop patterns by encoding fixes into repo docs/lints/tests.
If each developer's agents have different contexts/instructions, they produce inconsistent, conflicting code — the team coordination problem gets worse, not better.
✗ avoid: Letting each developer configure their own agent context independently without shared team conventions and quality gates.
— Maggie Appleton, GitHub, Ryan Lopopolo, OpenAI
When you're restructuring your engineering org for AI...
→ do: Push for 100% AI adoption — even 10% holdout creates outsized cultural drag. Pair your strongest engineers with AI agents first. Shift skill development toward spec writing, agent output review, and workflow design. Rethink sprint cadence — tasks that took days now take hours.
There's a 10x difference between 90% and 100% adoption. Strongest engineers benefit disproportionately and establish patterns for the rest.
✗ avoid: Forcing wholesale transformation overnight — experiment with AI-native team structures (smaller, more senior) alongside traditional teams first.
— Dan Shipper, Every, Mike Spitz, PFF, Martin & Natasha, McKinsey Software X
When you're hiring for AI engineering roles...
→ do: Evaluate candidates for AI fluency (ability to effectively use AI tools), not just traditional technical skills. Hire T-shaped engineers: broad AI literacy plus deep domain expertise. Seat AI teams close to product, not in isolated research groups.
The biggest hiring mistake is optimizing for ML research skills when you need ML engineering and production skills. A good SWE with AI fluency often outperforms a specialist without domain knowledge.
✗ avoid: Hiring pure AI researchers for production AI roles, or isolating the AI team from product teams.
— Denys Linkov, Wisedocs
When you're building review processes for AI-generated code...
→ do: Build custom linters and structural assertions (ESLint rules, deduplication checks, dependency edge validation). Deploy persona-based review agents (front-end architect, reliability engineer) triggered on every push, each primed with docs defining 'what good looks like.' Push plans as separate PRs with human review before execution.
Reviewing AI-generated code requires different techniques than human code, and human review alone can't keep up with agent output volume.
✗ avoid: Applying the same review process to AI-generated code as human-written code, or approving agent plans without explicit human review.
— Ryan Lopopolo, OpenAI, Martin & Natasha, McKinsey Software X
When you're building a startup with AI...
→ do: Embrace tiny teams (4-5 people can build what previously required 40-50). Use social-first distribution for consumer products. Take a portfolio approach: launch multiple small products, double down on winners. Aim for profitability early — inference cost collapse makes it viable.
AI tooling compresses the startup lifecycle: faster PMF but also faster scaling challenges. Capital efficiency is the new competitive advantage.
✗ avoid: Following the 2010s playbook of bloated teams, massive funding rounds, and slow iteration — AI makes each cycle nearly free.
— Sid Bendre, Oleve, Sarah Guo, Conviction

[tooling & mcp]

When you're building tools for your agent...
→ do: Design tool granularity to match the level of abstraction the model reasons at. Tools are just structured output — the model outputs JSON intent, your code executes it. Use bash as default tool; reserve structured tools for atomic, irreversible actions (sending email, payments).
Overly fine-grained tools (raw API calls) waste context; overly coarse tools hide important details the model needs.
✗ avoid: Wrapping every individual API endpoint as a separate tool, or building tools so coarse that the model can't express the action it needs.
— Luke Alvoeiro, Factory, Dex Horthy, HumanLayer, Thariq Shihipar, Anthropic
When you're integrating external tools and data sources...
→ do: Build MCP servers. If you're a tool/API provider, one MCP server covers all MCP-compatible AI apps. If you're building an AI app, make it an MCP client to access the entire ecosystem. Use the three control planes intentionally: tools (model-initiated), resources (app-managed with subscriptions), prompts (user-triggered).
MCP solves N×M → N+M: one integration instead of custom connectors for each AI app × each tool.
✗ avoid: Building custom one-off integrations for each AI application and tool pairing.
— Mahesh Murag, Anthropic, David Soria Parra, Anthropic

decision trees

10 hard questions builders face — answered with if/then logic from people who shipped
Should I use a single agent or multi-agent architecture?
[multi-agent]
Start single-agent. Go multi-agent only when: (1) context window is insufficient for the full task — subtasks need isolated contexts, (2) you need parallelism — independent subtasks can run concurrently, (3) you need specialization — different sub-tasks need fundamentally different tool sets or system prompts, or (4) you need adversarial evaluation — generator and evaluator must be separated. If none apply, a single agent with better context engineering will outperform a multi-agent system. Remember: 1 agent → 5 agents is building a distributed system, not adding features.
— Barry Zhang, Anthropic, Luke Alvoeiro, Factory, Sandipan Bhaumik, Ash & Andrew, Anthropic Applied AI
Should I use a framework or build a custom agent loop?
[agent design]
If you're prototyping or learning: use a framework (CrewAI, smolagents, LangGraph) to get to 70-80% fast. For production: own your loop. Most production agents are just a prompt + LLM call in a while loop with simple if statements. Frameworks get you started but trap you 7 layers deep when debugging prompt construction. The key test: can you see and control every token going to the model? If not, you'll hit a wall. Use the framework's ideas but implement in plain code you own.
— Dex Horthy, HumanLayer, Ryan Lopopolo, OpenAI, Thariq Shihipar, Anthropic
When should I use RAG vs fine-tuning vs both?
[rag]
Use RAG when: you need factual grounding in specific, changing documents; source attribution is required; or the knowledge base updates frequently. Use fine-tuning when: you need domain-specific style, format, or terminology; consistent behavior patterns; or edge deployment without retrieval infrastructure. Use both (the best approach for enterprise): fine-tune for domain style/format adaptation, RAG for factual grounding in specific documents. Never fine-tune to inject facts that could change — that's what RAG is for.
— Jerry Liu, LlamaIndex, Douwe Kiela, Contextual AI
Should I use bash/code-gen or structured tools for agent actions?
[tooling & mcp]
Default to bash/code execution — it's composable, low-context, and enables progressive discovery (pipe commands, jq, grep). Use structured tools only for: (1) atomic, irreversible actions (sending emails, making payments, writing files that need approval), (2) actions requiring strong type safety, or (3) operations where you need explicit permission gating. Code generation is most dynamic but needs linting/compilation verification. The spectrum: structured tools (reliable, high context) ↔ bash (composable, low context) ↔ code gen (dynamic, needs verification).
— Thariq Shihipar, Anthropic, Dex Horthy, HumanLayer
Orchestration or choreography for multi-agent coordination?
[multi-agent]
Use orchestration (central coordinator) when: tasks are sequential/dependent, you need global visibility and control, the workflow is relatively fixed, or you need to manage shared state. Use choreography (event-driven, pub/sub) when: agents are loosely coupled, you need independent scaling, workflows are dynamic/emergent, or you want to add/remove agents without changing the system. Most production systems use a hybrid: orchestration for the critical path, choreography for auxiliary/monitoring agents.
— Sandipan Bhaumik, Luke Alvoeiro, Factory
Should I build an agent or an automation workflow for this business process?
[agent design]
Use a deterministic workflow (DAG) when: the process is well-defined with clear steps, inputs/outputs are predictable, and reliability is paramount. Use an agent when: the process requires adapting to variable inputs, making judgment calls, or handling novel situations. Use a hybrid (recommended for most enterprise): DAG-based workflow with LLM nodes at decision points. Start with the DAG, selectively replace steps with LLM-driven components as you prove reliability at each step.
— Jerry Liu, LlamaIndex, Anju Kambadur, Bloomberg, Barry Zhang, Anthropic
When should I invest in building an MCP server vs custom integration?
[tooling & mcp]
Build an MCP server when: your tool/API will be used by multiple AI applications, you want ecosystem compatibility (Claude, ChatGPT, VS Code, Cursor), or you need both human-facing UI and agent-facing tools. Build a custom integration when: it's a one-off internal tool, you need tight coupling with a specific AI app's internals, or MCP's abstraction doesn't fit your interaction model. For most external-facing tools and APIs, MCP is the clear winner — build once, work everywhere.
— Mahesh Murag, Anthropic, David Soria Parra, Anthropic
How do I choose between assistive agents and automation agents?
[agent design]
Start with assistive agents (human-in-the-loop, producing drafts for review) for: high-stakes decisions, regulated domains, and when building organizational trust. Graduate to automation agents (background processing, minimal oversight) when: the task is repetitive and well-defined, error consequences are low/reversible, and you've validated accuracy through the assistive phase. The graduation path: assistive → semi-autonomous (agent acts, human approves) → fully autonomous (human audits samples).
— Jerry Liu, LlamaIndex, Anju Kambadur, Bloomberg
Should I use Opus-class or Sonnet-class models for this task?
[production]
Use Opus-class (highest capability, higher cost) for: planning, architecture decisions, complex multi-step reasoning, evaluating other agents' work, and tasks where quality >>> cost. Use Sonnet-class (strong capability, lower cost) for: implementation/execution of well-specified plans, routine tool calling, and high-volume operations. The optimal split: Opus for planning and evaluation, Sonnet for execution. This mirrors human teams where senior architects design and junior engineers implement.
— Ash & Andrew, Anthropic Applied AI
Should I build a vertical AI product or a horizontal platform?
[team & org]
Build vertical when: you have deep domain expertise, the domain has specific data/workflow requirements, and you can capture value through workflow integration. Build horizontal when: the capability is truly domain-agnostic, you can achieve distribution at scale, and the tooling gap is the primary bottleneck. The data says: the best AI startups are winning by being opinionated about the full stack for a vertical. Distribution is the moat, not model quality. Horizontal 'AI wrapper' dismissals are wrong — deep workflow integration creates real defensibility.
— Sarah Guo, Conviction, Clay Bavor, Sierra

implementation patterns

20 named patterns — what, when to use, and how to implement

[agent design]

Generator-Evaluator (GAN-inspired)
Separate builder and critic agents into independent context windows with different system prompts. The evaluator tests actual output (runs the app, uses Playwright) rather than reading diffs.
when: When agent self-evaluation is sycophantic, when tasks need quality verification, or when running long-duration agent tasks.
how: Spawn two agents: Generator gets the task + codebase context; Evaluator gets only the output artifact + acceptance criteria. Evaluator runs actual tests (Playwright, compilation, linting). If evaluation fails, Generator gets feedback in a new context. Never share conversation context between them.
— Ash & Andrew, Anthropic Applied AI
Ralph Loop (Fresh Context Iteration)
Break a prompt into discrete features, execute each in a fresh context window, loop until all pass tests. Uses JSON progress files as state between iterations.
when: Long-running tasks (1+ hours) where context rot degrades quality, or any multi-feature implementation project.
how: Initializer agent creates feature list + progress.json. Loop: fresh context → read progress → pick ONE unfinished feature → implement → test → git commit → update progress.json → repeat. 'Deterministically bad in an undeterministic world' — better to fail predictably than succeed unpredictably.
— Ash & Andrew, Anthropic Applied AI
LLM as Fuzzy Compiler
Treat code as a disposable build artifact compiled from context. The spec/requirements are the durable artifact; code is generated from constraints, conventions, and context by the LLM.
when: When building AI-native development workflows where specifications drive code generation.
how: Author specifications, architectural constraints, and conventions as the primary artifacts. Store in versioned context files (CLAUDE.md, specs/). LLM generates code from these constraints. Linters and tests serve as 'optimization passes' that validate the compiled output. Code is regenerable from context — specs are what you maintain.
— Ryan Lopopolo, OpenAI
Skills Architecture (Universal Agent + Modular Expertise)
One universal code-executing agent equipped with modular skill packages. Skills encode domain expertise (task instructions, tool configs, domain patterns). Code execution (bash, Python) is the universal action layer.
when: When you need agents across multiple domains and want to avoid building separate agent architectures for each.
how: Build one agent with code execution capability. Package domain expertise as skills (markdown instructions + tool configurations + reference examples). Use progressive disclosure: load skill metadata to decide which to activate, then load full instructions. Skills compose on top of MCP for tool access. Keep skills narrow (5-10 per domain) and deep.
— Barry Zhang & Mahesh Murag, Anthropic, Thariq Shihipar, Anthropic, Ryan Lopopolo, OpenAI
Harness-Model Co-evolution
The scaffolding harness doesn't disappear as models improve — it evolves. What was in the harness gets trained into the model, then the harness tackles new problems at the frontier.
when: When planning long-term agent architecture — design for the harness to evolve alongside model capabilities rather than becoming obsolete.
how: Build modular harnesses where each component can be selectively removed as models internalize that capability. Track which harness components are still necessary with each model generation. Invest in harness capabilities at the current model frontier (e.g., context management today, long-horizon planning tomorrow). Use first-party harnesses (Claude Code, Codex) when possible to benefit from post-training co-optimization.
— Ash & Andrew, Anthropic Applied AI, Ryan Lopopolo, OpenAI

[multi-agent]

Conductor-Drafter Architecture
Central Conductor agent creates plans and assigns tasks to specialized Drafter agents. Plans are living documents that get revised as Drafters discover new information.
when: Complex multi-step projects requiring task decomposition, parallel execution, and coordination across specialists.
how: Conductor breaks the project into a plan (living document). Assigns each task to a Drafter with the relevant context slice. Drafters execute and return results. Conductor reviews, revises the plan if needed, and assigns next tasks. Use file-system state for progress tracking and fresh context windows for each Drafter.
— Luke Alvoeiro, Factory
Agent Communication Bus
Agents communicate via typed events on a pub/sub bus rather than direct agent-to-agent calls, enabling loose coupling, independent scaling, and event sourcing for replay/debugging.
when: Multi-agent systems with 3+ agents that need loose coupling, independent deployment, or event-driven coordination.
how: Define typed event schemas for inter-agent communication. Agents publish events and subscribe to relevant topics. Use event sourcing to persist all events for replay and debugging. Implement circuit breakers for consumers and dead letter queues for unprocessable events. Use idempotent event handlers.
— Sandipan Bhaumik
Five-Pattern Multi-Agent Taxonomy
Five distinct coordination patterns: Delegation (parent→child, wait for response), Handoff (sequential pipeline), Broadcast (same task to many, aggregate), Orchestration (central coordinator), Autonomous Teams (peer-to-peer).
when: When designing multi-agent systems — use the taxonomy to select the simplest pattern that meets your coordination needs.
how: Delegation: for subtasks with clear input→output. Handoff: for sequential processing pipelines. Broadcast: for verification, voting, or diverse perspective generation. Orchestration: for complex projects needing central coordination. Autonomous Teams: for emergent collaboration (most complex, least predictable). Start with delegation; add complexity only when needed.
— Luke Alvoeiro, Factory

[memory & context]

Progressive Context Disclosure
Load only skill/tool metadata initially; expand full instructions only when the skill is activated. Agent discovers what it needs by browsing the file system rather than front-loading everything.
when: When the agent has many available skills/tools and front-loading all documentation would consume too much context window.
how: Store skills as directories with README frontmatter (trigger conditions, brief description) and detailed instructions in sub-files. Agent reads frontmatter to decide which skill to activate, then CD's into the directory and reads full instructions. Design CLIs with --help for the same pattern.
— Barry Zhang & Mahesh Murag, Anthropic, Thariq Shihipar, Anthropic, Ryan Lopopolo, OpenAI
Three-Layer Memory Architecture
Implement working memory (current session context), long-term memory (persistent cross-session storage in a database), and procedural memory (learned behavioral patterns encoded in skills/prompts).
when: Any agent that interacts with users across multiple sessions or needs to build persistent relationships.
how: Working memory: managed conversation context with compaction. Long-term memory: key-value or vector store with write-back loops after each interaction (decide what's worth persisting). Procedural memory: skills, few-shot examples, and behavioral patterns that encode HOW the agent works. Combine vector similarity (semantic 'what') with temporal ordering (episodic 'when') for recall.
— Richmond Alake, MongoDB

[evals & testing]

Eval Flywheel
Continuous improvement loop: production logs → human review → add interesting cases to eval datasets → improve offline evals → better prompts → better production → repeat.
when: Any AI feature in production. Start from day one and run continuously.
how: Instrument production with span-level tracing. Filter logs by low user feedback or automated scores. Have humans review flagged cases with a rubric. Add interesting failures and edge cases to the eval dataset. Run evals in CI/CD with threshold gates. Score individual pipeline steps, not just final output. Validate LLM-as-judge against human judgment periodically.
— Doug Guthrie, Braintrust

[production]

Swiss Cheese Security Defense
Layer three independent security mechanisms: model alignment (training-level safety), harness permissioning (AST parser blocking dangerous bash patterns), and sandboxing (network + filesystem isolation).
when: Any production agent that executes code, accesses the filesystem, or has network capabilities.
how: Layer 1: Use aligned models with safety training. Layer 2: AST-parse bash commands before execution, block patterns matching the 'lethal trifecta' (execute + filesystem + exfiltration). Layer 3: Run in isolated containers with network restrictions and filesystem boundaries. Error if agent writes to a file it hasn't read.
— Thariq Shihipar, Anthropic
Incremental Autonomy Ramp
Deploy agents on a trust gradient: copilot/assist mode → semi-autonomous (agent acts, human approves) → autonomous (human audits samples). Each stage requires proven reliability before advancing.
when: Any production agent deployment, especially in enterprise or high-stakes domains.
how: Stage 1: Agent suggests, human decides and acts. Stage 2: Agent acts, human reviews before commitment. Stage 3: Agent acts autonomously, human audits a sample of actions. Track reliability metrics at each stage; advance only when metrics exceed threshold. Build fallback paths at every stage.
— Anju Kambadur, Bloomberg, Jerry Liu, LlamaIndex
Token Budget Allocation (1/3-1/3-1/3)
Allocate roughly equal token budget across planning, implementation, and CI/verification rather than concentrating spend on code generation.
when: When budgeting compute costs for agent-driven development pipelines.
how: 1/3 on planning+specifications (tickets, architectural docs, plan review). 1/3 on implementation (code generation, tool calls). 1/3 on CI (linting, testing, review agent passes, verification). Track actual spend per category and rebalance. Spending tokens in CI is essential because getting code accepted is harder than writing it.
— Ryan Lopopolo, OpenAI

[rag]

Agentic RAG (Iterative Retrieval)
Retrieval becomes a tool the agent uses iteratively rather than a one-shot pipeline. The agent decides when to retrieve, what to retrieve, and whether retrieved context is sufficient before generating.
when: Multi-document reasoning, complex queries requiring multiple retrieval passes, or queries where the initial retrieval may be insufficient.
how: Give the agent a retrieval tool with query-reformulation capability. Agent retrieves, evaluates sufficiency, reformulates query if needed, retrieves again. For multi-document analysis: retrieve from each source, compare, synthesize. Add a 'confidence check' tool that lets the agent assess whether it has enough context before generating.
— Jerry Liu, LlamaIndex
End-to-End RAG Optimization
Optimize the entire RAG pipeline as a system rather than individual components in isolation. Retrieval quality and generation quality are deeply coupled.
when: When production RAG quality plateaus despite good individual component metrics.
how: Build joint evaluation: retrieval precision/recall AND generation faithfulness/relevance measured together. Tune chunk size/overlap, retrieval strategy (vector/BM25/hybrid), reranking, and generation prompts as a connected system. Use metadata filtering at scale. Set latency budgets for the full pipeline (30s+ causes user abandonment). Implement the 'I don't know' path for insufficient context.
— Jerry Liu, LlamaIndex, Douwe Kiela, Contextual AI

[team & org]

Garbage Collection Fridays
Dedicated weekly time where engineers categorically eliminate patterns that caused bad agent PRs by encoding fixes into repo docs, lints, tests, and structural assertions.
when: When using AI coding agents at scale and agent slop patterns keep recurring in code review.
how: Every Friday: review the week's agent-generated PRs that needed revision. Identify recurring patterns. For each pattern, create one of: custom ESLint rule, structural linter assertion, test case, or documentation update in CLAUDE.md/AGENTS.md. The goal is to make each class of error impossible to recur, not just fix individual instances.
— Ryan Lopopolo, OpenAI
Persona-Based Review Agents
Automated review agents triggered on every push, each primed with a specific reviewer persona (front-end architect, reliability engineer, security auditor) and docs defining 'what good looks like.'
when: When human review can't keep up with agent-generated code volume, or when you want consistent review across specific quality dimensions.
how: Create review agent personas, each with a system prompt defining their focus area and quality criteria docs. Trigger on every git push. Each persona reviews independently and posts comments. Human reviewers focus on the issues flagged by review agents rather than full code review.
— Ryan Lopopolo, OpenAI
Agent Onboarding as Documentation Test
Use agent onboarding as a proxy for documentation quality — if the agent can't figure out how to work in your codebase from your context files, neither can a new engineer.
when: When evaluating documentation quality, onboarding processes, or context engineering completeness.
how: Point a fresh agent (no prior context) at your repo with only the context files you'd give a new hire. Give it a representative task. If it succeeds, your documentation is good. If it fails, the failure points reveal exactly where your documentation gaps are. Fix the context, re-test with the agent.
— Patrick Debois, Tessl, Ryan Lopopolo, OpenAI

[tooling & mcp]

MCP + Computer Use Hybrid
Try structured MCP tool access first for interacting with services. Fall back to browser automation (computer use) for the long tail of systems without MCP endpoints.
when: When an agent needs to interact with a mix of MCP-enabled services and legacy web applications.
how: Check for .well-known/mcp.json on target services. If available, use MCP structured tools. If not, fall back to Playwright/Puppeteer browser automation. Build agents that can switch between modes seamlessly. As more services adopt MCP, the fallback path shrinks.
— Mahesh Murag, Anthropic
