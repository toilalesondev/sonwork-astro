How We Build Effective Agents

Barry Zhang, Anthropic

key insights

  • Progression: simple features → workflows (orchestrated multi-call pipelines) → agents (self-directed trajectories) → multi-agent collaboration. Broad trend: more agency = more useful but harder to control.
  • Don't build agents for everything — most tasks are better served by deterministic workflows. Agents shine only when the task requires dynamic decision-making based on environment feedback.
  • Keep it simple: start with the simplest scaffold possible. Augmented LLMs (model + tools + retrieval) solve most problems. Only add agentic loops when needed.
  • Think like your agent: trace through what the agent sees at each step. If YOU couldn't solve it given the same context, neither can the model. Empathy-driven prompt design.
  • Workflows vs agents distinction: workflows = predefined control flow trading cost/latency for performance; agents = model decides own trajectory based on environment feedback.
  • The next phase is agents becoming self-evolving — discovering and selecting their own tools dynamically rather than being pre-configured.

frameworks

[Augmented LLM (model + retrieval + tools)][Workflows → Agents → Multi-Agent spectrum][Think Like Your Agent (empathy-driven debugging)]

quotable

"If you can't solve the task given the same context your agent sees, your agent can't either."
"As we give these systems a lot more agency, they become more useful and also much harder to control."

→ what to do

  • Before building an agent, ask: does this task actually require dynamic decision-making, or would a deterministic workflow suffice?
  • Start with augmented LLM (model + tools + retrieval) — the simplest agentic pattern — and only add complexity when needed.
  • Debug by tracing exactly what context your agent sees at each decision point. If you couldn't solve it with that context, fix the context first.
  • Build agents that can discover and adopt new tools dynamically rather than hard-coding all capabilities upfront.

Don't Build Agents, Build Skills Instead

Barry Zhang & Mahesh Murag, Anthropic

key insights

  • Code is the universal interface — after building Claude Code, Anthropic realized a coding agent is actually a general-purpose agent. It can call APIs, manipulate file systems, analyze data with Python, and synthesize outputs.
  • Agent scaffolding can be as thin as bash + file system. The hard part isn't the agent loop — it's domain expertise.
  • Skills = reusable domain expertise packages. Instead of building separate agents per domain, build one universal code-executing agent and equip it with modular skills.
  • Progressive disclosure for context management: only load skill front-matter initially, expand full skill body only when instantiated. This preserves context window for actual work.
  • The paradigm shift: tighter coupling between model and runtime environment. Customization happens at the skill layer, not the agent architecture layer.
  • MCP became the standard for agent connectivity — skills build on top of MCP for tool access.

frameworks

[Skills Architecture (universal agent + modular domain skills)][Progressive Disclosure (front-matter → full skill body → code references)][Code as Universal Interface]

quotable

"We stopped building agents and started building skills instead."
"Code is not just a use case but the universal interface to the digital world."

→ what to do

  • Stop building separate agents for each domain. Build one code-executing agent and package domain expertise as reusable skills.
  • Use progressive disclosure: load only skill metadata into context initially, expand to full instructions only when the skill is activated.
  • Design skills as composable expertise modules: task instructions, relevant tool configurations, and domain-specific patterns bundled together.
  • Leverage code execution (bash, Python, file system) as the universal action layer rather than building custom tool integrations for each domain.

12-Factor Agents

Dex Horthy, HumanLayer

key insights

  • Most production agents don't use frameworks — they're just code with an LLM call in a loop. Frameworks get you to 70-80% but then you're 7 layers deep in a call stack debugging prompt construction.
  • Not every problem needs an agent. If you find yourself specifying exact steps in the prompt, you should have written a bash script.
  • 12 factors distilled from 100+ builder interviews: (1) Natural language → structured output, (2) Own your prompt, (3) Own your context window, (4) Tools are just structured output, (5) Unify execution regardless of trigger, (6) Keep prompts small and focused, (7) Own your control flow with simple code, (8) Contact on model capabilities not providers, (9) Compact/summarize aggressively, (10) Prompt-engineer with small changes, (11) Evals are just unit tests, (12) Fail loudly and visibly.
  • Tools are just structured output — the model outputs a JSON blob saying 'I want to call X with args Y', your code executes it. There's no magic.
  • Own your control flow: don't let the framework decide when/how the model gets called. Use simple while loops and if statements.
  • Evals are just unit tests with a different assertion strategy. Don't over-complicate them.

frameworks

[12-Factor Agents (inspired by 12-Factor App methodology)][Tools as Structured Output pattern][Own Your Loop (explicit control flow over framework abstractions)]

quotable

"I spent two hours adding more and more detail to the prompt until I realized I could have written the bash script in 90 seconds. Not every problem needs an agent."
"Most production agents are not built with frameworks. They are just a prompt with an LLM call in a loop."

→ what to do

  • Own your prompt template — don't let a framework construct it for you. You need to see and control every token going to the model.
  • Own your context window — explicitly manage what goes in. Compact/summarize aggressively to avoid context rot.
  • Keep control flow in simple code (while loops, if statements) rather than framework abstractions. You need to debug this.
  • Treat tools as structured output: model returns intent as JSON, your code executes it deterministically.
  • Build evals as unit tests from day one — input/expected-output pairs with LLM-as-judge for fuzzy assertions.
  • Fail loudly: surface errors visibly rather than silently retrying or swallowing exceptions.

The Multi-Agent Architecture That Actually Ships

Luke Alvoeiro, Factory

key insights

  • The bottleneck in software engineering is no longer intelligence — it's human attention. Engineers can only drive a few tasks forward per day because each requires their supervision.
  • Five frontier multi-agent patterns: (1) Delegation — parent spawns child, waits for response. (2) Handoff — sequential pipeline, each agent passes to next. (3) Broadcast/fan-out — same task to multiple agents, aggregate results. (4) Orchestration — central coordinator dispatching to specialists. (5) Autonomous teams — agents communicate peer-to-peer without central coordinator.
  • Factory's architecture uses 'Drafter Agents' (workers) coordinated by a 'Conductor' (orchestrator). Conductor breaks tasks into plans, assigns to drafters, reviews outputs, iterates.
  • Key insight on planning: the plan is a living document, not a static spec. Agents must be able to revise plans as they learn more about the codebase.
  • Tool granularity matters enormously: overly fine-grained tools (individual API calls) waste context; overly coarse tools hide important details. Find the right abstraction level.
  • For tasks requiring hours of work: use file-system state, progress tracking, and fresh context windows per sub-task to avoid context degradation.

frameworks

[5-Pattern Multi-Agent Taxonomy (Delegation, Handoff, Broadcast, Orchestration, Autonomous Teams)][Conductor-Drafter Architecture][Living Plan pattern (plan as mutable artifact)]

quotable

"The bottleneck in software engineering nowadays is not intelligence. It's now limited by human attention."
"What if a human decides what to build and then a system figures out how to do so?"

→ what to do

  • Match your multi-agent pattern to the task: use delegation for simple subtasks, orchestration for complex multi-step projects, broadcast for verification/consensus.
  • Implement plans as living documents that agents can revise as they discover new information about the problem.
  • Design tool granularity carefully — tools should match the level of abstraction the model reasons at, not raw API endpoints.
  • For long-running multi-agent tasks: use file-system state persistence, progress tracking files, and fresh context windows per sub-task.
  • Start with a single well-prompted agent, then decompose into multi-agent only when you hit clear bottlenecks in context, capability, or parallelism.

Collaborative AI Engineering

Maggie Appleton, GitHub

key insights

  • The 'one man, two dozen Claudes' theory is flawed. Software is a team sport — scaling individual output doesn't solve coordination problems, it makes them worse.
  • Implementation is rapidly becoming a solved problem. The hard question is no longer 'how to build it' but 'should we build it?' — agreeing on what to build is the actual bottleneck.
  • Current AI coding tools are all single-player interfaces. They scale the individual but ignore the team coordination problem.
  • Spec-driven development becomes critical: agents need unambiguous specifications. This forces teams to resolve disagreements upfront rather than during implementation.
  • The ratio of planning to implementation is inverting: expect 80% planning, 20% coding (opposite of today). Writing specs IS the engineering work.
  • Agent alignment problem at the team level: if each developer has their own agents with different contexts/instructions, the agents will produce inconsistent, conflicting code.
  • Need shared context layer: team-wide conventions, architectural decisions, and constraints that ALL agents can access — not just individual developer preferences.

frameworks

[Spec-Driven Development (specifications as primary engineering artifact)][Collaborative AI Engineering (team-level agent coordination)][Planning-First Ratio Inversion (80% planning, 20% implementation)]

quotable

"Believing individual productivity leads to great software is 'nine women make a baby in one month' logic."
"The hard question is no longer how to build it, it's should we build it? Agreeing on what to build is the actual bottleneck."

→ what to do

  • Invest in shared specification documents that serve as the source of truth for both humans and agents.
  • Build shared context layers (architectural decision records, conventions, constraints) accessible to all agents on the team.
  • Shift engineering effort toward planning and specification writing — this becomes the primary skill as implementation gets automated.
  • Don't just scale individual developer output with more agents; solve the coordination problem first with shared specifications and team-level agent alignment.
  • Treat agent-generated code like code from a new team member: it needs the same architectural guardrails and review processes.

Architecting Agent Memory

Richmond Alake, MongoDB

key insights

  • The evolution: chatbots → RAG → reasoning/tool-use → AI agents. Memory is what transforms stateless AI apps into stateful, relationship-building agents.
  • Three memory types for agents: (1) Short-term/working memory — current conversation context. (2) Long-term memory — persistent facts, preferences, past interactions stored in a database. (3) Procedural memory — learned patterns of HOW to do things, encoded in prompts/tools.
  • Memory reduces prompt engineering burden: instead of stuffing context into system prompts, store and retrieve it dynamically from persistent memory.
  • Memory enables relationship-building: agents remember user preferences, past interactions, and context across sessions — transforming transactional interactions into ongoing relationships.
  • Semantic memory (vector search) + episodic memory (time-stamped interaction logs) should be combined for effective recall. Use vector similarity for 'what' and temporal ordering for 'when'.
  • Memory architecture must be first-class: treat it like a database schema design problem, not an afterthought bolted onto the prompt.

frameworks

[Three-Layer Memory Architecture (Short-term, Long-term, Procedural)][Semantic + Episodic Memory combination][Stateless-to-Stateful Agent transformation]

quotable

"We're turning the responses in our AI applications and making our agents build relationships with our customers — and all of it is centered around memory."
"Asking 'what is an agent' is like asking 'what is consciousness' — it's a spectrum."

→ what to do

  • Design agent memory as a first-class architectural concern: define schemas for what to remember, how to store it, and when to recall it.
  • Implement at minimum two memory layers: working memory (current session context) and long-term memory (persistent cross-session storage).
  • Combine vector similarity search (semantic memory) with time-stamped logs (episodic memory) for richer recall.
  • Use memory to reduce prompt engineering: store domain knowledge, user preferences, and interaction patterns in persistent storage rather than cramming everything into system prompts.
  • Build memory write-back loops: after each interaction, decide what's worth persisting for future sessions.

Long-Running Agents

Ash & Andrew, Anthropic Applied AI

key insights

  • Three core challenges for long-running agents: (1) Context — windows are finite, context rot degrades coherence, context-sense anxiety makes models rush near the end. (2) Planning — models try to one-shot everything or build half-features. (3) Self-evaluation — models are sycophantic about their own output, can't judge quality.
  • Model + harness co-evolution: from Sonnet 3.7 (1hr minimal scaffold) to Opus 4.6 (12hrs). The harness doesn't disappear as models improve — it evolves. What was in the harness eventually gets trained into the model, then the harness tackles new problems.
  • Generator-Evaluator pattern (inspired by GANs): separate builder and critic into different context windows with different system prompts. Evaluator uses Playwright to actually test the app, not just read diffs. Crucial: don't share context between generator and evaluator — let the evaluator judge only outputs.
  • Ralph Loop technique: break prompt into features, execute each in a fresh context window, loop until all pass tests. 'Deterministically bad in an undeterministic world' — better to fail predictably than succeed unpredictably.
  • Long-running harness architecture: Initializer agent creates feature list + progress file as JSON (models overwrite markdown but not JSON). Loop: fresh context → get bearings → run init script → pick ONE unfinished feature → implement → test with Playwright → git commit → update progress → repeat.
  • Key primitive advances: checkpoints (code rewind), server-side compaction (indefinite runs), skills with progressive disclosure, programmatic tool calling (write code to batch tool calls, return only final result), agent teams (sub-agents communicate peer-to-peer, not just through parent).
  • Reading traces by hand is the most important debugging skill. Empathize with the model — close your eyes and try to navigate a webpage opening them every 10 seconds. That's what the model experiences.

frameworks

[Generator-Evaluator Pattern (GAN-inspired adversarial agent loop)][Ralph Loop (iterative fresh-context execution)][Harness-Model Co-evolution][Long-Running Agent Harness (initializer → feature loop → verification)]

quotable

"A year ago, Claude was struggling to write bash commands. Now almost all of Claude Code is being written by Claude Code, and it can run effectively for days."
"It's better to fail predictably than to succeed unpredictably."

→ what to do

  • Implement Generator-Evaluator pattern: separate builder and critic agents with independent context windows. Evaluator should test actual output (Playwright, puppeteer) not read diffs.
  • Use JSON files for persistent state (feature lists, progress tracking) — models are less likely to overwrite JSON than markdown.
  • Give each sub-task a fresh context window rather than running everything in one growing session. Use file-system state to maintain continuity.
  • Build verification loops: after implementing a feature, have the agent actually run it, test it visually, and verify before marking complete.
  • Read agent traces line-by-line — there is no substitute for understanding what the model actually sees and thinks at each step.
  • Use Opus-class models for planning and Sonnet-class for execution to balance cost vs capability.
  • Don't share context between generator and evaluator — let the evaluator judge outputs independently to avoid the model 'kidding itself'.

Harness Engineering: How to Build Software When Humans Steer and Agents Execute

Ryan Lopopolo, OpenAI

key insights

  • Implementation is no longer the scarce resource. Code is free. The scarce resource is defining work, prioritizing it, and effectively empowering agents to complete it
  • Banned his team from touching editors — forced all work through agents (Codex). Result: each engineer has access to 5-5,000 engineers worth of capacity
  • The harness should surface instructions to the model at the right time, not front-load everything. Defer context until lint/test time so the agent prototypes first, then refines
  • Post-training happens in the context of the harness (apply_patch tool, bash tool quoting). Depending on first-party harnesses gives you leverage from this co-training
  • Code is a disposable build artifact. The spec/requirements are the durable artifact, code is compiled from context by the LLM as a 'fuzzy compiler'
  • Garbage Collection Fridays: every engineer spends one day/week categorically eliminating patterns that caused bad PRs, closing the loop between human review feedback and agent context
  • Structure repos like a 10,000-engineer org even if you're small: 750 packages, isolated by domain, one way to do everything. Makes agent output more predictable regardless of where it looks
  • Every time you type 'continue' to the agent is a failure of the harness to provide enough context for autonomous completion
  • Review agents replace human reviewers: spin up persona-based review agents (front-end architect, reliability engineer) that run on every push against docs that define 'what good looks like'
  • Token budget split: roughly 1/3 planning+tickets, 1/3 implementation, 1/3 CI. Spending tokens in CI is essential because getting code accepted is harder than writing it
  • Skills should be narrow (5-10) and deep rather than wide. Infrastructure beneath skills changes frequently — hide complexity so humans and agents don't track churn
  • If you use plan mode, push plans as separate PRs with human review before execution — unapproved plans encode bad instructions

From Chaos to Choreography: Patterns for Multi-Agent Systems

Sandipan Bhaumik

key insights

  • Going from 1 agent to 5 agents is not adding features — it's building a distributed system. You inherit all distributed systems problems: race conditions, coordination, state management, failure recovery
  • Two coordination patterns: Orchestration (central controller routes work, good for sequential/dependent tasks) vs Choreography (event-driven, agents react to events independently, better for scalability and loose coupling)
  • State management in multi-agent systems requires treating agent state like database transactions — use event sourcing and checkpointing to maintain consistency
  • Design for failure: assume agents WILL fail. Implement circuit breakers, dead letter queues, retry with exponential backoff, and graceful degradation
  • The Agent Communication Bus pattern: agents publish/subscribe to typed events rather than calling each other directly, enabling loose coupling and independent scaling
  • Idempotency is critical: every agent operation should be safely retryable without side effects. Design agent tools to be idempotent by default

frameworks

[Orchestration vs Choreography patterns for agent coordination][Agent Communication Bus (pub/sub typed events)][Event Sourcing for agent state management][Circuit Breaker pattern for agent failure recovery][Dead Letter Queue for failed agent messages]

quotable

"They think adding more agents is just like adding more features. It's not. It's building a distributed system."
"One agent works beautifully. Product comes back wanting five more, and here's what happens: you have coordination, you have state, you have failure modes you never imagined."

→ what to do

  • Treat multi-agent systems as distributed systems from day one. Apply distributed systems patterns: event sourcing, circuit breakers, idempotency
  • Choose orchestration for dependent/sequential workflows, choreography for independent/scalable ones. Many production systems use a hybrid
  • Implement checkpointing and event sourcing for agent state so you can replay, debug, and recover from failures
  • Make every agent operation idempotent. Use dead letter queues for messages that agents repeatedly fail to process
  • Use an event bus for agent communication rather than direct agent-to-agent calls — this enables independent deployment and scaling

Building Agents with MCP Workshop

Mahesh Murag, Anthropic

key insights

  • MCP = Model Context Protocol. Inspired by LSP (Language Server Protocol). Standardizes how AI apps interact with external systems via three interfaces: tools (model-controlled), resources (application-controlled), prompts (user-controlled).
  • MCP solves the N×M integration problem: without it, every AI app builds custom integrations to every tool/data source. With MCP, build your server once → works with every MCP client.
  • Three interface types with distinct control planes: Tools = model decides when to invoke. Resources = application decides when to attach (can be dynamic, support notifications/subscriptions). Prompts = user-triggered templates for common interactions.
  • OAuth 2.0 support added to protocol: enables remote servers (SSE transport), where server handles auth handshake with end application (e.g., Slack), holds the token, gives client a session token.

  • Well-known endpoint pattern (.well-known/mcp.json): websites can advertise their MCP capabilities, complementing computer use — agent tries structured MCP first, falls back to browser automation for the long tail.
  • Resources support subscriptions/notifications: server can notify client when a resource changes, enabling event-driven agent architectures.
  • Composability: MCP servers can also be MCP clients, enabling hierarchical agent systems where orchestrator agents delegate to specialist agents, all communicating via the protocol.
